技术部开发规范 v2.0

一、总则

本规范适用于技术部所有软件开发项目，旨在统一代码风格、提高代码质量、降低维护成本。所有开发人员必须遵守本规范，Code Review 时以此为评审依据。

二、项目结构规范

所有项目必须遵循以下目录结构：

backend/
  ├── app/
  │   ├── api/          # API 路由层
  │   ├── core/         # 业务逻辑层
  │   ├── models/       # 数据模型层
  │   ├── utils/        # 工具函数
  │   └── main.py       # 应用入口
  ├── tests/            # 测试用例
  ├── requirements.txt  # 依赖清单
  └── .env.example      # 环境变量模板

前端项目使用 Vue 3 + Vite，必须开启 TypeScript 严格模式。

三、代码规范

3.1 命名规范
- 变量/函数：snake_case（Python）、camelCase（JavaScript）
- 类名：PascalCase
- 常量：UPPER_SNAKE_CASE
- 私有方法：以单下划线开头 _private_method
- 禁止使用拼音命名，禁止使用 a/b/c 等无意义变量名

3.2 函数规范
- 单个函数不超过 50 行
- 单个函数参数不超过 5 个
- 必须添加类型注解（Python）或 TypeScript 类型
- 复杂逻辑必须写注释说明

3.3 注释规范
- 公共 API 必须有 docstring
- 注释说明 WHY 而不是 WHAT
- TODO/FIXME/HACK 标记必须注明责任人和日期

四、Git 工作流

4.1 分支规范
- main：生产分支，只接受 merge，不直接 commit
- develop：开发分支
- feature/xxx：功能分支，从 develop 切出
- hotfix/xxx：紧急修复分支，从 main 切出
- release/x.x.x：发布分支

4.2 Commit 规范
采用 Conventional Commits 格式：
- feat: 新功能
- fix: 修复 Bug
- docs: 文档更新
- refactor: 代码重构
- test: 测试相关
- chore: 构建/工具相关

示例：feat: 新增用户登录接口，支持 JWT 认证

4.3 Code Review
- 所有代码合并到 main/develop 前必须经过 Code Review
- 至少一名同级或以上开发人员审批
- Review 关注：逻辑正确性、安全性、性能、可维护性

五、数据库规范

5.1 设计原则
- 所有表必须有主键 id（UUID 或自增整数）
- 所有表必须有 created_at 和 updated_at 时间戳
- 外键关系必须在数据库层面定义
- 禁止使用数据库保留字作为表名或字段名

5.2 索引规范
- 高频查询字段必须建索引
- 联合索引遵循最左前缀原则
- 定期分析慢查询日志并优化

5.3 安全规范
- 严禁 SQL 拼接，必须使用参数化查询
- 敏感字段（密码、手机号）加密存储
- 生产数据库操作需经过审批

六、API 设计规范

6.1 RESTful 风格
- GET /api/v1/users — 获取用户列表
- GET /api/v1/users/{id} — 获取单个用户
- POST /api/v1/users — 创建用户
- PUT /api/v1/users/{id} — 更新用户
- DELETE /api/v1/users/{id} — 删除用户

6.2 响应格式
统一返回格式：
{
  "code": 0,
  "message": "ok",
  "data": { ... }
}

错误码规范：
- 0：成功
- 400：参数错误
- 401：未认证
- 403：无权限
- 404：资源不存在
- 500：服务器内部错误

6.3 安全规范
- 所有接口必须验证用户身份（JWT Token）
- 敏感接口需增加二次验证
- 请求频率限制：普通接口 100次/分钟，敏感接口 10次/分钟
- 响应中不得暴露服务器信息、数据库结构等敏感内容

七、测试规范

7.1 测试覆盖率要求
- 核心业务逻辑：不低于 80%
- 工具函数：不低于 90%
- 整体项目：不低于 70%

7.2 测试分类
- 单元测试：测试单个函数或方法
- 集成测试：测试模块间交互
- 端到端测试：测试完整用户流程

7.3 测试命名
test_功能描述_预期结果

八、部署规范

8.1 环境划分
- dev：开发环境，开发人员自测
- staging：预发布环境，产品/测试验收
- production：生产环境

8.2 部署流程
1. 代码合并到 release 分支
2. 自动触发 CI/CD 构建
3. 运行测试套件（全部通过方可继续）
4. 构建 Docker 镜像
5. 部署到 staging 环境验收
6. 验收通过后部署到 production

8.3 回滚策略
- 每次部署保留最近 3 个版本的镜像
- 出现重大问题时，5 分钟内回滚到上一稳定版本
- 回滚后必须进行根因分析并记录

九、性能规范

9.1 接口响应时间要求
- 查询接口：P99 < 200ms
- 写入接口：P99 < 500ms
- 文件上传：P99 < 3s

9.2 数据库查询
- 禁止 SELECT *，必须指定字段
- 单次查询返回记录数不超过 1000 条
- 复杂查询使用 EXPLAIN 分析执行计划

9.3 缓存策略
- 高频读取、低频变更的数据使用 Redis 缓存
- 缓存过期时间根据业务场景设置
- 缓存穿透/击穿/雪崩需有防护方案

十、日志与监控

10.1 日志规范
- 使用结构化日志（JSON 格式）
- 日志级别：DEBUG < INFO < WARNING < ERROR < CRITICAL
- 生产环境最低日志级别为 INFO
- 敏感信息（密码、Token）不得出现在日志中

10.2 监控指标
- 接口响应时间、错误率
- 数据库连接池状态
- 内存/CPU 使用率
- 磁盘使用率
- 业务指标（用户活跃度、功能使用率等）

10.3 告警策略
- 接口错误率 > 5%：立即告警
- 接口响应时间 P99 > 1s：告警
- 磁盘使用率 > 85%：告警
- 内存使用率 > 90%：告警
